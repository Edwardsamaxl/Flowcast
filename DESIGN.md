# 流转 - 产品设计文档

## 0. 当前设计决策

本版对齐最新 PRD 和当前讨论结论。

| 项目 | 当前决策 |
|---|---|
| 产品入口 | `/` 是 Landing Page，不是工作台。 |
| 应用内页面 | 只有 3 个：资产库、流转工作台、历史。 |
| 产品主叙事 | 帮助创作者把一份内容流转到更多平台，而不是只做三个平台的文案生成。 |
| MVP 可用链路 | 当前先展示并实现“视频 -> 文字”。 |
| 前端预留 | 音频、图文、文本入口和远期流转方向要设计出来，但可标注稍后接入。 |
| 命名 | 全站使用“创作者画像”，不再主打“声纹画像”。 |
| 平台范围 | 前端必须体现多平台覆盖感：小红书、抖音、B 站、视频号、公众号、知乎、X、快手、微博、即刻等。 |
| 资产写入 | 上传或拆解后先展示画像变化，由创作者确认后写入。 |
| 流转工作台 | 不只是上传内容，还要覆盖转写解析、用户确认、画像更新确认、多平台输出和反馈改稿。 |
| 历史页面 | 支持按创作者筛选：全部、具体创作者、无画像。 |
| 悬浮球 | 删除产品内不明用途悬浮操作。 |

## 1. 产品视觉定位

流转不是普通 AI 写作页，也不是单平台运营工具。它更像一个“内容流转中控台”：创作者把已有内容放进来，系统解析内容、识别表达方式，再把它改写成不同平台能发布的版本。

界面要传达三种感受：

- **流动**：从上传到解析、确认、沉淀、输出，用户能看见内容正在跨平台转换。
- **可控**：创作者始终知道哪些内容只是用于当前稿件，哪些内容会写入长期画像。
- **扩展**：当前先做视频转文字，但用户能看见未来可流转到音频、图文、文本、视频脚本等形态。

## 2. Landing Page 设计

Landing Page 是产品介绍页，目标是让用户快速理解：

- 流转能把一份内容扩展到更多平台。
- 当前 MVP 先开放视频转文字。
- 创作者画像让内容更像本人。
- 用户可以从 Landing 直接进入创建画像或流转工作台。

### 2.1 首屏

主文案：

> 让视频、音频、图文、文本自由流转。

副文案：

> 把长视频、直播回放、课程录屏、口播、图文和文章，转成适合不同平台发布的内容版本。当前先开放视频转文字链路，并通过创作者画像尽量保留本人的观点、语气和表达方式。

首屏 CTA：

- 创建创作者画像
- 拆解一条内容

首屏必须包含一个“流转链路”视觉：

```text
内容上传 -> 转写解析 -> 用户确认 -> 创作者画像沉淀 -> 多平台输出
```

这个链路要比普通功能卡更重要。它应该是首屏的核心视觉，而不是一个小补充。

### 2.2 平台展示

平台展示不能只写小红书、知乎、X。要突出跨平台分发能力，至少包含：

- 小红书
- 抖音
- B 站
- 视频号
- 公众号
- 知乎
- X
- 快手
- 微博
- 即刻

平台展示建议用横向平台带、平台矩阵或输出样张。不要写成“支持平台”小标签后就结束；用户需要感知“我熟悉的平台可以进来，我不熟的平台也能出去”。

### 2.3 页面结构

1. 顶部导航
   - 左侧：流转 Flowcast Logo。
   - 中间：流转路径、平台覆盖、创作者画像、常见问题。
   - 右侧：可不放按钮，避免和首屏 CTA 打架。

2. Hero
   - 小标签：创作者内容跨平台流转工具。
   - 主文案：让视频、音频、图文、文本自由流转。
   - 副文案说明当前先开放视频转文字。
   - 两个 CTA。
   - 首屏视觉展示“内容上传 -> 转写解析 -> 用户确认 -> 创作者画像沉淀 -> 多平台输出”。

3. 多平台输出示例
   - 同一段视频分别转成小红书、抖音、B 站、公众号、知乎、X 等不同表达。
   - 每个平台展示短样张，不展示完整长文。

4. 像本人写的对比
   - 左侧：普通 AI 文案。
   - 右侧：调用创作者画像后的版本。
   - 对比点聚焦语气、判断、禁用表达、常用结构。

5. 创作者画像
   - 展示画像包含：定位、观点、语气、案例、禁用表达、平台规则。
   - 强调解析后展示变化，创作者确认后写入。

6. 远期流转方向
   - 展示音频、图文、文本、视频脚本等入口。
   - 标注当前状态：视频转文字可用，其它稍后接入。

7. 常见问题
   - 当前为什么先做视频转文字。
   - 上传内容是否自动改画像。
   - 流转任务是否默认沉淀。
   - 是否支持抖音、B 站、公众号等平台。

### 2.4 Landing Page 不应出现

- 历史任务列表。
- 复杂工作台状态。
- 付费定价结构。
- 虚假的自动发布承诺。
- 暗示已经支持完整视频生成、图片生成或自动剪辑。

## 3. 应用内结构

应用内只保留三页：

1. 资产库
2. 流转工作台
3. 历史

应用内品牌副标题建议从“表达资产档案室”改成更贴近主叙事的：

- 内容流转工作台
- 创作者流转中控台
- 跨平台内容工作台

## 4. 资产库页面设计

资产库目标：管理“当前在记录谁的创作者画像”。

页面不应像普通文件仓库，而应该像“创作者画像 + 画像素材证据”。

推荐结构：

```text
创作者列表 / 新建
  + 当前创作者画像
  + 画像素材与更新建议
```

### 4.1 创作者管理

- 当前创作者。
- 创作者列表。
- 新建创作者。
- 切换创作者。
- 每个创作者显示画像完整度、最近更新时间、素材数量。

### 4.2 创作者画像

画像内容包括：

- 创作者名称。
- 人设定位。
- 领域。
- 语气特征。
- 高频观点。
- 常用案例。
- 常用句式。
- 标题偏好。
- 常用结构。
- 禁用表达。
- 平台写法规则。

视觉上要像“创作者档案”，而不是普通表单堆叠。平台规则是画像的一部分，不应被放成毫无上下文的独立卡片。

### 4.3 画像素材与更新建议

素材列表不应叫“源文件列表”，建议命名为：

- 画像素材
- 画像证据
- 可沉淀素材

每个素材展示：

- 素材标题。
- 输入类型：视频 / 音频 / 图文 / 文本。
- 来源平台或来源类型。
- 解析状态。
- 是否已写入画像。
- 可写入候选数量。

写入确认必须展示 diff：

- 将新增的观点。
- 将补充的案例。
- 将修改的平台规则。
- 将新增的禁用表达。
- 将忽略的内容。

主操作：

- 预览画像变化。
- 写入当前创作者画像。
- 忽略本次解析。

## 5. 流转工作台页面设计

流转工作台是核心工作流页面。它不是单纯的“上传内容”页面，而是一次内容流转任务的完整操作台。

页面目标：把一份内容转成多个平台版本，并让用户决定是否把其中有价值的内容沉淀进创作者画像。

### 5.1 页面主流程

必须突出以下流程：

```text
内容上传 -> 转写解析 -> 用户确认 -> 创作者画像沉淀 -> 多平台输出
```

建议以顶部步骤条或左侧流程轨呈现。不要让用户只看到一堆上传框、下拉框和编辑器。

工作台至少要覆盖四种工作模式：

1. **上传与解析**：上传内容、查看转写、查看观点/案例/金句。
2. **画像更新确认**：查看系统建议如何改变创作者画像，并决定写入、忽略或手动编辑。
3. **多平台稿件**：查看和编辑各平台输出。
4. **反馈改稿**：通过标签和自然语言继续要求 LLM 修改稿件或更新画像。

### 5.2 左侧：内容输入与解析

左侧负责“这份内容说了什么”。

输入类型：

- 视频：MVP 可用。
- 音频：入口预留，稍后接入。
- 图文：入口预留，稍后接入。
- 文本：入口预留，稍后接入。

内容展示：

- 上传或选择已有素材。
- 解析状态。
- 转写摘要。
- 转写全文（默认折叠）。
- 核心观点。
- 案例。
- 金句。
- 可沉淀画像候选。

### 5.3 右侧：配置、输出、反馈与画像确认

右侧顶部是任务配置：

- 创作者画像选择。
- 目标平台多选。
- 是否允许进入画像候选。

平台选择必须体现多平台：

- 小红书。
- 抖音。
- B 站。
- 视频号。
- 公众号。
- 知乎。
- X。
- 快手。
- 微博。
- 即刻。

右侧主体采用工作模式切换：

1. **画像更新确认**
   - 展示本次解析会对当前创作者画像造成哪些变化。
   - 将变化分成：新增、修改、建议忽略。
   - 用户可以选择单条接受、单条忽略、手动编辑后接受。
   - 明确显示写入目标创作者和来源素材。
   - 主按钮为“确认写入创作者画像”。
   - 如果用户只想临时流转内容，可以选择“不写入，仅用于本次任务”。
2. **稿件**
   - 平台版本切换。
   - 可编辑稿件。
   - 保存草稿。
   - 复制。
   - 重新生成。
3. **反馈**
   - 标签反馈。
   - 自然语言反馈。
   - 只影响当前稿件。
   - 建议写回创作者画像。

这个结构比单纯上下堆叠更适合现在的信息量。

画像更新确认不能只是稿件页里的一个提示框。它必须是工作台内一个明确可进入的页面/模式，因为用户需要在这里判断：

- 这条内容是否代表创作者长期观点。
- 哪些观点、案例、句式、禁用表达值得写入。
- 哪些只是本次视频里的临时表达，不应污染长期画像。
- 平台规则是否需要更新。

### 5.4 工作台状态

流转工作台需要展示清晰状态，而不是只显示“处理中”：

- 待上传。
- 转写中。
- 解析中。
- 待确认画像更新。
- 已确认画像更新。
- 生成多平台稿件中。
- 待反馈。
- 已保存到历史。

## 6. 历史页面设计

历史页目标：查找、复盘、继续修改。

必须支持按创作者筛选：

- 全部。
- 具体创作者。
- 无画像。

可选筛选：

- 输入类型。
- 输出平台。
- 是否有反馈。
- 是否已沉淀到画像。
- 状态。

列表展示：

- 任务标题。
- 源素材。
- 输入类型。
- 创作者画像。
- 输出平台。
- 反馈状态。
- 沉淀状态。
- 最近更新时间。

详情页展示：

- 源素材信息。
- 转写文本。
- 使用的创作者画像。
- 使用的平台规则。
- 生成稿件。
- 反馈记录。
- 写回画像记录。
- 继续修改入口。

详情页不是另一个完整工作台，重点是复盘和追溯；继续修改时可跳回流转工作台。

## 7. 视觉方向

视觉方向应建立在当前项目已有的温白纸面、墨色文字和 `seal` 强调色上，但不能只停留在“纸面档案室”。新版产品更像“创作者内容流转工作台”，所以视觉上要同时有编辑台的可信感和流转路径的方向感。

当前色彩口径：

- 页面底色延续温白纸面：`#fffdf8`，适合承载长文本、转写、稿件编辑和画像档案。
- 主文字延续墨色：`#201b17`，保持中文内容工具的阅读稳定性。
- 当前强调色 `#b4472f` 可继续作为 `seal` 使用。它接近朱砂/印泥色，但在产品里不应被包装成复古中式风格，而是作为“确认、写入、流转当前节点”的操作强调色。
- 辅助背景使用浅纸色和边线，不做大面积彩色块，避免变成营销 SaaS 风。
- 平台覆盖感主要通过平台矩阵、输出样张和流转线表达，不依赖给每个平台上品牌色。

界面气质：

- Landing Page：更开放，强调内容跨平台扩散，有横向流转、平台矩阵和输入输出形态。
- 资产库：更稳，像创作者画像档案，重点是可追溯、可确认、可编辑。
- 流转工作台：更强操作感，突出步骤、状态、模式切换和左右协同。
- 历史：更像检索和复盘界面，低装饰、高可扫读。

可以增加：

- 横向流转线。
- 平台矩阵。
- 输入/输出形态切换。
- 轻量状态标签。
- 画像更新 diff 预览。
- 工作台模式 tabs。
- 当前流程节点标记。

避免：

- 蓝紫渐变。
- 抽象 AI 光效。
- 复杂数据仪表盘。
- 装饰性水墨、印章、复古纹理。
- 只用三个功能卡解释产品。
- 把远期能力设计得像已经完全可用。

## 8. 文案规范

优先使用：

- 创作者画像。
- 内容流转。
- 多平台输出。
- 画像候选。
- 写入当前创作者画像。
- 只影响当前稿件。
- 影响后续生成。
- 预览画像变化。

避免作为主文案使用：

- 声纹画像。
- 一键爆款。
- 全自动分发。
- AI 赋能创作。
- 释放创作潜能。
- 下一个内容革命。

“声纹”可以作为底层能力或旧概念偶尔出现，但界面和文档主命名统一为“创作者画像”。

## 9. 验收清单

- [ ] Landing Page 主叙事是跨平台内容流转，不是三平台文案生成。
- [ ] Landing Page 展示“内容上传 -> 转写解析 -> 用户确认 -> 创作者画像沉淀 -> 多平台输出”。
- [ ] 首屏 CTA 分别进入资产库和流转工作台。
- [ ] 平台展示包含小红书、抖音、B 站、视频号、公众号、知乎、X、快手、微博、即刻等。
- [ ] 页面展示用“创作者画像 / 创作者资产”，不再主打“声纹档案”。
- [ ] 资产库展示画像变化预览，用户确认后写入。
- [ ] 流转工作台有输入类型入口：视频、音频、图文、文本。
- [ ] 流转工作台有工作模式切换：上传与解析、画像更新确认、稿件、反馈。
- [ ] 历史支持按创作者筛选，包含“无画像”。
- [ ] 应用内导航只有资产库、流转工作台、历史。
- [ ] 远期入口可展示，但不可暗示已完整实现。
