# 流转 - 产品设计文档

## 0. 设计决策锁定

本版设计先对齐 PRD 第 11 节的待讨论问题，作为后续界面实现的默认口径。

| 问题 | 当前决策 | 设计影响 |
|---|---|---|
| 首页主文案 | 采用方案 A：让视频内容自然流转。 | 首页保持产品入口气质，不做营销型夸张表达。 |
| 资产库上传视频默认行为 | 上传后只解析，用户确认后再写入人物画像。 | 资产库必须有“解析预览 -> 用户确认 -> 写入画像”的明确状态链路。 |
| 拆解内容是否默认沉淀进资产库 | 默认不沉淀。 | 拆解页面需要提供显式按钮/开关，让用户选择是否把该视频沉淀到对应人物资产库。 |
| 右侧输出和对话是否臃肿 | 暂按方案 A：输出区和对话区上下排列。 | 右侧保持一列纵向工作流：配置、输出、对话反馈，不引入抽屉或 tabs。 |
| 多人物画像 MVP 处理 | 当前只做一个人物，但允许创建多个，UI 预留下拉选择。 | 不把人物写死；所有涉及人物的页面都使用人物选择器，并预留“新建人物”入口。 |

额外产品前提：

- MVP 重点验证“视频能否很好转化成人设资产”。
- 资产库视频用于建立或更新人物画像，必须强调“可控写入”。
- 拆解内容默认只服务当前稿件，除非用户主动选择沉淀。

## 1. 设计方向

**设计关键词：中文创作者档案室、视频流转工作台、可信的表达画像。**

流转不是泛用 AI 写作工具，也不是内容运营看板。它更像一个为中文知识创作者准备的“表达资产档案室”：用户把视频、直播、课程录屏放进来，系统先转写、解析、提取观点和表达习惯，再由用户确认哪些内容可以成为某个人物的长期画像资产。

界面需要同时呈现三种感受：

- **可控**：用户始终知道哪些内容只是被解析，哪些内容已经写入人物画像。
- **像本人**：人物画像不是普通表单，而是一份可编辑的人物表达档案。
- **可流转**：同一条视频可以拆成小红书、知乎、X 等平台内容，但不会默认污染人设库。

一句话判断：用户打开后应觉得“这是我的表达资产库和视频流转工具”，而不是“又一个 AI 生成按钮”。

## 2. 视觉系统

### 2.1 风格基调

采用 **Editorial Structuralism**：出版编辑感 + 精密软件结构。

- 背景用温白、浅墨、纸面灰，避免冷蓝灰和营销 SaaS 渐变。
- 重要状态用朱砂红，不大面积铺色。
- 页面像档案台、批注台、稿件台，而不是卡片堆叠。
- 信息层级靠排版、留白、细线、纸面分区建立，不依赖重阴影。

### 2.2 颜色 Token

| Token | Value | 用途 |
|---|---:|---|
| `paper-0` | `#fffdf8` | 页面主背景 |
| `paper-50` | `#f8f3ea` | 次级背景、侧栏 |
| `paper-100` | `#efe7d9` | 分区底色、表头 |
| `paper-200` | `#ded2bf` | 强分割线 |
| `ink-950` | `#201b17` | 主标题、强正文 |
| `ink-800` | `#3f352d` | 正文 |
| `ink-600` | `#75685a` | 辅助说明 |
| `ink-400` | `#a79a8a` | 占位、弱标签 |
| `seal-50` | `#f8e8df` | 选中底色、轻提示 |
| `seal-500` | `#b4472f` | 主操作、当前状态、关键批注 |
| `seal-600` | `#963824` | 主操作 hover |
| `sage-500` | `#6f7d59` | 已确认、已写入 |
| `amber-500` | `#b9822e` | 解析中、待确认 |
| `red-500` | `#a6382e` | 错误、失败 |

使用规则：

- 朱砂红只用于主 CTA、当前导航、待确认写入、关键批注。
- 不使用蓝紫渐变、玻璃拟态、大面积饱和色。
- 状态必须配文字，不只依赖颜色。
- 资产写入类操作要比普通保存更醒目，因为它会影响后续生成。

### 2.3 字体

| 角色 | Font Stack | 用途 |
|---|---|---|
| Display / Page Title | `"Noto Serif SC", "Songti SC", "SimSun", serif` | 品牌、首页主文案、人物档案标题 |
| UI / Body | `"Noto Sans SC", "PingFang SC", "Microsoft YaHei", sans-serif` | 导航、按钮、表单、正文 |
| Mono / Data | `"JetBrains Mono", "SFMono-Regular", Consolas, monospace` | 编号、时间、转写片段定位 |

排版规则：

- 首页 H1：48-64px，宋体或衬线，行高 1.08。
- 页面标题：34-44px，行高 1.12。
- 区块标题：20-24px。
- 正文与稿件编辑区：15-16px，行高 1.7。
- 标签、状态、编号：12px。
- 中文标题使用 `text-wrap: balance`，正文使用 `text-wrap: pretty`。

## 3. 信息架构

一级导航保留四个页面：

1. 首页
2. 资产库
3. 拆解内容
4. 历史

删除或不单独出现：

- 编辑台
- 独立校准页
- 不明用途悬浮球

“校准”只作为生成后的编辑、反馈、写回动作，出现在“拆解内容”和“历史详情”中。

## 4. 全局布局

桌面端使用左侧导航 + 主工作区：

- 左侧导航宽度：248-264px。
- 主工作区最大宽度：1360px。
- 页面左右边距：32-48px。
- 主区纵向间距：32px 起。

移动端：

- 单列布局。
- 一级导航改为底部导航或顶部紧凑菜单。
- 资产库、拆解内容的左右布局折叠为上下流。
- 任何选择器、按钮、反馈输入的触控高度不低于 44px。

## 5. 页面设计

### 5.1 首页

首页是产品入口，不是工作台。

主文案固定为：

> 让视频内容自然流转。

副文案：

> 建立人物声纹资产库，让 AI 学会你的观点、语气和表达方式。无论是长视频还是直播内容，都能转化成像本人写的多平台文字内容。

核心 CTA：

- 创建声纹资产库
- 拆解一条内容

首页结构：

1. **Hero 区**
   - 左侧：主文案、副文案、两个 CTA。
   - 右侧：一个“视频 -> 人物画像 -> 多平台稿件”的编辑样张。
   - 不展示工作台统计，不展示历史任务。

2. **流转示例**
   - 展示同一段视频如何转成小红书、知乎、X。
   - 每个平台只展示一小段，不做完整文章。
   - 强调“平台写法不同，但观点来源一致”。

3. **像本人写的对比**
   - 左侧为普通 AI 文案。
   - 右侧为调用人物画像后的版本。
   - 对比点聚焦语气、判断、禁用表达、常用结构。

首页不出现：

- 最近任务列表。
- 数据卡片仪表盘。
- 复杂设置。
- 任何不明用途浮动按钮。

### 5.2 资产库

资产库目标：管理“当前在记录谁的声纹”。

页面采用左右布局。

左侧：源文件与解析区。

- 上传视频。
- 展示已上传源文件。
- 展示状态：待解析、转写中、已解析、待确认写入、已写入画像、失败。
- 点击源文件后展示转写文本、核心观点、语气特征、可写入候选项。
- 上传后的默认动作是解析，不自动写入人物画像。

右侧：人物声纹档案。

- 顶部必须有人物选择器。
- 当前 MVP 只有一个默认人物也要使用下拉选择。
- 允许创建多个人物，但不做复杂管理。
- 不把人物名称写死在 UI 或数据结构中。

人物档案包含：

- 人物名称。
- 人设定位。
- 领域。
- 语气特征。
- 高频观点。
- 常用案例。
- 常用句式。
- 标题偏好。
- 常用内容结构。
- 禁用表达。
- 平台写法规则。

资产写入流程：

```text
上传视频
  -> 系统转写
  -> 系统解析观点、语气、结构、禁用表达候选项
  -> 用户查看解析预览
  -> 用户选择写入哪些内容
  -> 用户确认写入当前人物画像
  -> 画像更新，并保留来源视频记录
```

关键控件：

- `解析该视频`
- `预览可写入内容`
- `写入当前人物画像`
- `忽略本次解析`
- `新建人物`
- `切换人物`

写入确认必须明确显示：

- 即将写入的人物。
- 写入的内容类型。
- 来源视频。
- 是否覆盖旧规则，还是追加为新候选。

### 5.3 拆解内容

拆解内容是核心工作流页面。

目标：用户上传或选择一条视频，选择人物画像和目标平台，然后得到可编辑、可反馈的多平台文字内容。

页面保持左右布局。

左侧：视频输入与内容解析。

- 拖入视频。
- 选择已有源文件。
- 展示转写状态。
- 展示转写文本。
- 展示核心观点、案例、金句、段落。

右侧：配置、输出、对话反馈上下排列。

1. **配置区**
   - 人物画像选择器。
   - 默认可选择“无画像”。
   - 可选择已创建人物。
   - 平台选择：小红书、知乎、X。
   - 沉淀设置默认关闭。

2. **输出区**
   - 平台版本切换。
   - 生成稿件编辑器。
   - 保存草稿。
   - 重新生成。
   - 复制。

3. **对话反馈区**
   - 标签反馈。
   - 自然语言反馈。
   - 只影响当前稿件。
   - 可选择将本次反馈写回人物画像。

沉淀规则：

- 拆解内容默认不沉淀进资产库。
- 页面提供明确按钮：`沉淀到当前人物资产库`。
- 如果未选择人物画像，该按钮不可用，并提示先选择或创建人物。
- 用户点击后进入确认态，展示将沉淀的观点、案例、句式、语气特征。
- 沉淀成功后，该视频在历史中标记为“已沉淀”。

右侧布局暂按方案 A：

```text
配置区
输出区
对话反馈区
```

不使用抽屉，不使用 tabs。后续如果右侧空间过重，再评估方案 B 或 C。

### 5.4 历史

历史记录每一次拆解任务。

列表展示：

- 源文件标题。
- 使用的人物画像。
- 输出平台。
- 生成状态。
- 是否有反馈。
- 是否已沉淀到资产库。
- 最近更新时间。

详情展示：

- 源视频信息。
- 转写文本。
- 使用的人物画像。
- 使用的平台规则。
- 生成稿件。
- 反馈记录。
- 对话式改稿记录。
- 写回人物画像的记录。

历史详情里的反馈同样需要区分：

- 只影响当前稿件。
- 写回人物画像，影响后续生成。

## 6. 核心组件

### 6.1 人物选择器

所有涉及人物画像的位置都使用统一人物选择器。

状态：

- 无画像。
- 当前人物。
- 新建人物。
- 多人物下拉列表。

MVP 只有一个人物时，下拉仍然存在，但内容可以是：

- 无画像
- 默认人物
- 新建人物

### 6.2 资产写入确认面板

用于资产库和拆解内容的沉淀确认。

必须展示：

- 写入目标人物。
- 来源视频。
- 候选写入内容。
- 写入方式：追加、覆盖、忽略。
- 确认 CTA。

这是高风险动作，不能隐藏在普通保存按钮里。

### 6.3 视频解析状态

状态文案：

- 待解析
- 转写中
- 已解析，待确认
- 已写入画像
- 已沉淀到资产库
- 解析失败

不要使用“处理中”作为唯一状态，必须说明正在做什么。

### 6.4 按钮

- Primary：主流程推进，例如创建资产库、生成稿件、确认写入。
- Secondary：普通操作，例如复制、保存草稿、重新生成。
- Destructive / Risk：覆盖画像、删除来源、放弃解析。
- Quiet：查看详情、展开转写、收起。

按钮动效：

- Hover 使用轻微背景变化。
- Pressed 使用 `transform: translateY(1px)` 或 `scale(0.98)`。
- 动画使用自定义 cubic-bezier：`cubic-bezier(0.32, 0.72, 0, 1)`。

### 6.5 卡片与面板

主要面板采用双层结构：

- 外层：浅纸面托盘，细边线，8-16px padding。
- 内层：内容纸面，轻微内高光，层级清晰。

避免：

- 大面积玻璃拟态。
- 重阴影。
- 过圆的卡片。
- 三列等高功能卡。

## 7. 内容语气

文案要像可信的编辑工具，不像营销页。

推荐：

- 解析视频
- 预览可写入内容
- 写入当前人物画像
- 沉淀到当前人物资产库
- 只影响当前稿件
- 影响后续生成
- 来源视频
- 表达画像
- 禁用表达

避免：

- 一键打造爆款
- 释放创作潜能
- 智能赋能创作者
- 开启下一代内容革命
- 全自动孵化 IP

## 8. 实现注意事项

- 使用现有 Next.js + Tailwind，不迁移框架。
- Tailwind token 以 `paper / ink / seal / sage / amber` 为主。
- 所有中文文件必须保持 UTF-8。
- 不再写入乱码中文。
- 新增组件前优先复用现有 `AppShell`、`SectionTitle`、`TextareaBlock`、`IconPill` 等结构。
- 人物相关数据不得写死具体人物名称。
- 拆解内容的沉淀动作必须是显式用户操作。
- 资产库上传后不得自动更新画像，必须先解析再确认。

## 9. 验收清单

- [ ] 首页主文案为“让视频内容自然流转。”
- [ ] 首页只展示入口、解释、示例和对比，不展示工作台状态。
- [ ] 资产库上传视频后只解析，不自动写入画像。
- [ ] 资产库有明确的“确认写入人物画像”流程。
- [ ] 拆解内容默认不沉淀进资产库。
- [ ] 拆解内容有“沉淀到当前人物资产库”按钮或开关。
- [ ] 右侧输出和对话采用上下排列。
- [ ] 当前只做一个人物，但 UI 允许创建多个并预留下拉选择。
- [ ] 所有人物选择处都不写死具体人物。
- [ ] 历史记录能看出任务是否已沉淀到资产库。
